basics

  • h16
  • ls32
  • pct-encoded: we can simply check if a component ptr is inside uri text to tell if that component is normalized
  • pchar = unreserved / pct-encoded / sub-delims / ":" / "@"
  • segment = *pchar
  • segment-nz = 1*pchar
  • segment-nz-nc = 1*( unreserved / pct-encoded / sub-delims / "@" )
  • URI = scheme ":" hier-part [ "?" query ] [ "#" fragment ]
  • URI-reference = URI / relative-ref
  • relative-ref = relative-part [ "?" query ] [ "#" fragment ]
    relative-part diff with hier-part only between path-noscheme VS path-rootless, the former is defined as `segment-nz-nc *( "/" segment )`
    A relative reference that begins with two slash characters is termed a network-path reference
    relative reference that begins with a single slash character is termed an absolute-path reference.
    A relative reference that does not begin with a slash character is termed a relative-path reference.
  • absolute-URI: URI without the fragment component; we can invent the concept relative-URI

scheme

scheme = ALPHA *( ALPHA / DIGIT / "+" / "-" / "." ) where ALPHA is caseless; if a buf does not starts with a scheme, it's not a URI(but may be a relative-ref)

branches of hier-part

  1. "//" authority path-abempty: if a URL's hier-part begins with "//", an authority(may be an empty authority) follow it(path-absolute begins with a '/' and a pchar, path-rootless begins with a pchar)
  2. path-absolute: if scheme is "data"(zxxu: archive file specifying data in it), in theory we can figure out a URI which has its hier-part as path-absolute(note file:///path/to/your/file belong to the first branch which has authority as empty)
  3. path-rootless: some URI don't have a concept of path or root-path
  4. path-empty: such as "mailto:" is valid and will open a blank email client

authority( in hier-part 01 )

paths

path-abempty = *( "/" segment )
path-absolute = "/" [ segment-nz *( "/" segment ) ]
path-rootless = segment-nz *( "/" segment )

query

query = *( pchar / "/" / "?" )

fragment

fragment = *( pchar / "/" / "?" )

The fragment identifier component of a URI allows indirect identification of a secondary resource, so if an URI is malicious, it's still malicious after removed the Fragment component

Suffix Reference

has the same syntax as a relative-path reference; cannot be used in contexts where a relative reference is expected. As a result, suffix references are limited to places where there is no defined base URI
